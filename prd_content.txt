1. Executive Summary
TimberFlow is a quotation-first, full-stack Sales and Operations SaaS platform purpose-built for timber suppliers and dealers in India. It replaces disconnected spreadsheets, WhatsApp quotes, and paper ledgers with a single integrated system covering the entire business lifecycle — from inquiry to dispatch to GST filing.
The system is designed specifically for the Indian timber trade: CFT/CBM-based pricing, HSN code mapping, CGST/SGST/IGST split invoicing, e-Way bill integration, and GSTR-1/3B export — all built in from the ground up, not bolted on.

2. Problem Statement
Indian timber suppliers currently operate with fragmented, error-prone workflows:
Quotations are created manually on WhatsApp or paper, with no version control, no follow-up tracking, and no conversion pipeline to invoices.
Inventory is managed in physical registers or basic Excel sheets with no real-time visibility into stock levels, low-stock alerts, or species-wise valuation.
GST compliance is a major pain point — CGST/SGST vs IGST logic (intra-state vs inter-state), HSN code mapping for timber species, GSTR-1 preparation, and ITC reconciliation are all done manually and error-prone.
Customer receivables are tracked informally — there is no aging analysis, no automated follow-up, and outstanding dues are often forgotten.
There is no single view connecting a customer inquiry to quotation, order, dispatch, invoice, and payment — each step lives in a different place.

9. Functional Requirements
9.1 GST Engine
This is the most critical business logic in the system. All GST calculations must be deterministic and auditable.
Determine transaction type based on supplier state vs customer state:
Same state (intra-state): CGST = rate/2, SGST = rate/2, IGST = 0
Different state (inter-state): IGST = full rate, CGST = 0, SGST = 0
GST rates by HSN: 4407 (sawn timber) = 12%; 4403 (logs) = 0%; 4409 (mouldings) = 18%. All rates configurable per item.
Tax is calculated on each line item, then summed — never on the grand total directly.
All tax amounts rounded to 2 decimal places using banker's rounding.
GSTR-1 export must produce valid JSON per GSTN schema (B2B, B2C, CDNR, CDNUR invoice types).
ITC (Input Tax Credit) from purchase invoices is tracked separately and shown in GSTR-3B summary.

9.3 Invoice Rules
Invoices generated from quotations inherit all line items; editable only in DRAFT status.
Once marked SENT, only payment recording is allowed — no line item edits.
Invoice PDF must include: company GSTIN, customer GSTIN, HSN codes, tax rate, CGST/SGST/IGST amounts, grand total in words, e-Way bill number if applicable.
Invoice number sequence: INV-[FY]-[NNNN] — e.g., INV-2425-0089. Resets each financial year (April 1).
CANCELLED invoices generate a Credit Note automatically.
9.4 Stock Management Rules
Stock is decremented when a dispatch order is created (status: LOADING), not when invoice is raised.
Stock is incremented when a purchase invoice is recorded against a supplier.
All stock movements are recorded in StockLedger with reference ID for full traceability.
Low-stock alert is triggered when currentStock < minStockLevel; visible on dashboard and items list.
Negative stock is blocked by default; configurable override for OWNER role.
9.5 Receivables & Payables
Customer outstanding = sum of unpaid/partial invoices. Aging buckets: 0-30, 31-60, 61-90, 90+ days.
Supplier payable = sum of purchase invoices where payment is pending.
Partial payments are recorded against specific invoices with date, amount, and mode (cash/NEFT/UPI/cheque).
Credit limit enforcement: system warns (not blocks) when a new invoice would breach credit limit.

15.2 CGST/SGST vs IGST Decision Logic
The place of supply rules under GST determine whether a transaction is intra-state or inter-state:
Supplier state = Customer's billing/shipping state → Intra-state → CGST + SGST
Supplier state ≠ Customer's billing/shipping state → Inter-state → IGST only
Exports → Zero-rated → IGST at 0% (LUT) or with refund claim
Use customer's state code (stored on Customer record) against org's state code (stored on Organisation) to determine automatically on every transaction.
